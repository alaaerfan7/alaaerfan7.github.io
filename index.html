Overview

How I like to approach security work

Understand the system, experiment in a safe environment, then design controls that are realistic for the people who have to live with them.

Before adding more tools, I try to understand how the current network, app or team is actually behaving. That might mean diagramming flows, looking at packet captures, or walking through how a user really experiences a login page or alert.

From there, I like building small, focused things: a detection rule, a Python script, a lab topology, or a new way of visualising logs. The goal is always the same: make it easier to see what is happening, and easier to respond when something is wrong.

Featured project

ARP cache poisoning detection & prevention

MSc dissertation project: building and evaluating a system that spots ARP spoofing in virtual networks and automatically updates the network to block the attack.

The aim was to go beyond ‚Äúlog and alert‚Äù and actually change how the network behaves when a spoofing attempt is detected. The system ties together packet inspection, anomaly detection and SDN-based mitigation.

  • Monitors ARP traffic with Scapy and flags suspicious IP/MAC changes.
  • Talks to a Ryu SDN controller to install or update flow rules in real time.
  • Acts as a hybrid ARP proxy / watchdog so legitimate traffic still works.
  • Includes a custom ARP attack tool to test edge cases and detection coverage.
  • Evaluated with different tools and scenarios inside an Open vSwitch lab.
